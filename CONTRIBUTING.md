# Contributing

Thank you for your interest in contributing to Subtributary!

Subtributary focuses on the tools used to create websites, so feedback from people actually using the software is especially valuable.
Documentation improvements are contributions too!
If something was difficult to understand or set up, we would appreciate help making it easier for the next person.

## Reporting issues

Github issue templates are available for common types of reports, including bug reports and feature requests.
Using the appropriate template helps us understand the problem and respond more quickly.

If you're unsure whether something is a bug or simply expected behavior, feel free to ask.

## General feedback

Not every suggestion needs to become a GitHub issue.

If you have general thoughts about the projects, usability, documentation, or ideas for the future, we'd love to hear about them through our community channels.

## Contributing code

Pull requests are welcome.

Please use the provided pull request template and keep changes focused on a single topic whenever practical.
Smaller pull requests are generally easier to review and merge than large, unrelated changes.

Before investing significant time in implementing a new feature,
consider opening a feature request first so we can discuss whether it aligns with the project's direction.

## Project priorities

Subtributary exists to build simple, high-quality software that works well for people around the world.
When making design decisions, we generally prioritize the following:

1. Internationalization — Software should be designed so that it can be localized easily. Features should work across languages, scripts, writing directions, and cultures whenever practical.
2. Accessibility — Features should follow accessibility best practices and strive to be usable by as many people as possible.
3. Simplicity — Readable code, maintainable designs, and clear APIs are generally favored over clever or highly abstract implementations.

These priorities are intended to guide discussion rather than replace engineering judgment.
Every project has trade-offs, and thoughtful conversations are always welcome.

## Using AI tools

Responsible AI-assisted development is welcome.

Contributors are responsible for understanding everything they submit, verifying that it is correct, and ensuring they have the right to contribute it under the project's license.
